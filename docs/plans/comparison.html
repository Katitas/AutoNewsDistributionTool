リアルタイム ニュース検索方式 比較表

AutoNewsDistributionTool ― カチタス実需中古住宅向け 不動産ニュース自動配信

作成日: 2026-05-15 ／ 対象規模: 月30起動 × 1起動あたり最大50検索（=月1,500クエリ） ／ 配信時刻: JST 08:00

本ドキュメントの目的

Bedrock の search_real_news ツールが内部で叩く Web Search API について、現行候補（Brave / Tavily）に加えて Google・Bedrock Deep Research などを含めた 9 種類 をフラットに比較する。 観点は「カチタスの不動産ニュース配信で実際に効くか」に絞り、机上の理想ではなく月3,000円未満の運用予算で月30回安定配信できるかを軸に判定する。

前提となる利用パターン:
① 1日1回 08:00 JST に Lambda 起動 → ② Bedrock Opus 4.7 が agentic loop で 5〜50 回の Web 検索を実行 → ③ 5件のニュースを抽出し SES + Slack に配信。 検索クエリは日本語中心、対象は「不動産・PropTech・法規制・市況・リフォーム」、リアルタイム性は 過去24〜72時間のニュースで十分。

① 比較対象の選定

ユーザー指定の4方式（Tavily / Brave / Google / Bedrock Claude Opus Deep Research）に、同等以上のティアで Lambda から実用可能な5方式を加えた計9方式を比較する。

# 方式 カテゴリ 選定理由 / なぜ同等ティアか

除外した方式（参考）
  • You.com API — 英語中心で日本語ニュースの取得品質が他と比べ顕著に劣る。
  • Bing Web Search API — 2025年8月で公式終了済み（Microsoft）。Azure AI Foundry の Grounding with Bing に統合されたが、AWS 構成からは外部依存が増える。
  • GDELT Project — 完全無料だが、ニュースの「機械翻訳された見出し+トーン分析」が主体で、本文要約や日本語の原文取得には不向き。研究用途。
  • DuckDuckGo Instant Answer / Kagi — Kagi は B2C 向けで API は限定的、DuckDuckGo はニュース検索が貧弱。
  • RSS 直接購読 — リアルタイム性は最高だが、対象ソース選定とパースが運用負担に。本ツールの「広く拾う」要件と合致しない。

③ 各方式の特徴と本ツールでの位置づけ

① Brave Search API 採用中

  • 強み: 無料2000q/月で本ツール想定の月1500qを完全カバー。/news/search エンドポイントが NewsBox 相当の結果を返し、freshness=pd で過去24時間のフィルタが可能。独自インデックスのため Google ToS の影響を受けない。
  • 弱み: 申請時にクレジットカード登録が必須（無料枠でも）。返却される description が短い（200文字程度）ため、Bedrock 側で本文を再取得しないと要約品質が落ちる。
  • 本ツールでの位置づけ: 無料運用の主軸として最適。現行実装の PROVIDER_DEFAULT_CAPS で 50/起動の上限制御済み。

② Tavily Search API 代替済み

  • 強み: LLMエージェント前提に設計され、include_raw_content=true で本文要約済みコンテンツが返却される。Bedrock の二段階呼び出し（検索→本文再取得）が不要になり、エージェントループが短縮される。日本語の要約品質が他の汎用検索より一段上。
  • 弱み: 無料1000q/月で本ツール想定の月1500qを下回るため、上限25/起動に抑える必要がある。リクエスト1回あたりのレイテンシが Brave の2-3倍。
  • 本ツールでの位置づけ: 品質優先時の切替先。プロンプト改善で済む場合は Brave 据え置きで良いが、要約品質に不満が出たら切替候補。

③ Google Custom Search JSON API

  • 強み: 日本語ニュースサイトの網羅性は世界最強。siteRestrict で住宅新報・日経不動産等を直指定でき、dateRestrict=d1 + sort=date で時系列ソート可能。
  • 弱み: 無料100q/日（=月3,000q）だが、1日に複数回テスト実行するとすぐ枯渇する。1,000q ごと $5 の従量課金は妥当だが、Google Programmable Search Engine 経由のため ニュース専用エンドポイントが存在しない（Google News API は2011年に廃止済み）。News Box 結果の構造化は SerpAPI に劣る。
  • 本ツールでの位置づけ: 単独採用は微妙。Brave/Tavily の補完として site:指定の専門ニュース検索に使うのが良い。

④ Bedrock + Claude Opus Deep Research（AgentCore Browse Tool）

  • 強み: AWS純正、IAM完結、追加API契約・キー管理不要。Bedrock AgentCore の Browse Tool（2025年GA）で Claude が自律的にブラウジング・検索ループを実行。Deep Research プロンプトを使えば1回の呼び出しで「複数ソースを横断調査→構造化レポート出力」まで完結する。
  • 弱み: Browse Tool は内部的に Bedrock が選定した Web 検索プロバイダ（Brave 等）を呼び出すため、結局裏で外部APIが動く。コスト構造が「Opusトークン + 内部検索」の二重課金で読みにくい。Deep Research モードは1回の探索で10-60秒、$0.5-2 程度かかり、5件のニュース取得には過剰スペック。
  • 本ツールでの位置づけ: 過剰。本ツールは「決まったカテゴリで5件取得」というルーチン作業であり、Deep Research の自律探索能力は活かしきれない。月次レポートのような単発の深掘り調査向き。

⑤ Perplexity Sonar API

  • 強み: 引用付きの完成された回答を一発で取得できる。Sonar Pro は Deep Research と同等の探索深度を持ちつつ、料金体系が $5/1000リクエスト（固定）+ トークン課金で予算が読みやすい。日本語ネイティブ対応。
  • 弱み: Bedrock からの呼び出しは Tool Use 経由になるため、AWS純正の世界観から外れる。Sonar Pro の出力は「Perplexityの編集視点」を経由するため、ニュース原文へのこだわりがある場合は不向き。
  • 本ツールでの位置づけ: 第3候補。Brave/Tavily で品質に行き詰まったら検討。

⑥ Exa (旧 Metaphor) API

  • 強み: 埋め込みベースのセマンティック検索により「リフォーム需要の高まりを示すニュース」のような意味検索が効く。includeText / excludeText でフィルタが柔軟。highlights で本文の関連部分のみ抽出可能。
  • 弱み: 日本語の埋め込み精度が英語より落ちる。Exa は「お宝記事を掘る」のは得意だが、「今日のメインストリームニュース」を拾うのは Brave/Google に劣る。
  • 本ツールでの位置づけ: PropTech・海外不動産カテゴリ専用の補助検索として混在採用が筋。

⑦ SerpAPI

  • 強み: Google検索結果の News Box / Top Stories を構造化JSONで取得できる。Google CSE では取れない「Google ニュース」タブ相当の結果が手に入る。
  • 弱み: 本質的に Google結果のスクレイピングプロキシであり、Google ToS との関係がグレー。月100q無料はテスト用にすら足りず、本番運用は $75/月（5000q）からで本ツール予算（月3,000円目安）を圧迫する。
  • 本ツールでの位置づけ: 不採用。コスト・規約リスクが見合わない。

⑧ NewsAPI.org

  • 強み: ニュース特化型の老舗、対応ソース150,000、日本のNHK・日経・朝日等もカバー。sources パラメータで配信元を絞り込める。
  • 弱み: 無料プランは 開発用途のみ・本番運用禁止（ToS明記）。商用本番は最低 Business プラン $449/月。返却される description が極端に短く、本文取得には別途スクレイピングが必要。
  • 本ツールでの位置づけ: 不採用。ToS違反のリスクと本番コストが見合わない。

⑨ Anthropic Web Search Tool（Claude API native）

  • 強み: {"type": "web_search_20250305"} を tools に追加するだけで Claude が自律検索を実行。引用付きの完成回答が返り、Lambda 側の実装はほぼゼロ。allowed_domains / blocked_domains でドメイン制御も可能。
  • 弱み: Bedrock 経由では2026年5月時点で未対応（Anthropic直APIのみ）。本プロジェクトは Bedrock 採用のため、Anthropic API への切替が必要になり、IAM完結・データ主権の AWS 統合メリットを失う。料金は $10/1000検索 + トークン課金で、月1500検索なら約 $15+トークン代。
  • 本ツールでの位置づけ: 条件付き有望。Bedrock 経由でサポートされた段階で再評価する価値あり（要ウォッチ）。

⑤ 最終推奨

結論: 現行の Brave Search API（フォールバック=Tavily）を維持

根拠（3点）

  1. コスト ― 月1,500クエリは Brave 無料枠の75%に収まる。他の有料候補（Sonar $7.5/月、SerpAPI $75/月、NewsAPI $449/月）に乗り換える経済合理性がない。
  2. 品質 ― カチタスの不動産ニュース配信は「広く拾って Bedrock Opus 4.7 が選別」する設計のため、検索API側に高度な要約は不要。Tavily のような LLM特化型は provider 切替で即時試せる現行アーキテクチャが優れている。
  3. 規約・運用安全性 ― Brave/Tavily ともに公式API、ToS明確、ハード上限が存在する。SerpAPI のスクレイピング系・NewsAPI の本番利用制限のような潜在的リスクを抱えない。

将来の見直しトリガー

  • Bedrock 経由で Anthropic Web Search Tool(⑨) がGAされたら → 実装コードが大幅に削減できるため再評価。
  • 住宅新報・R.E.port 等の 専門メディア取得率が低下したら → Google CSE(③) の siteRestrict 補助検索を追加。
  • 要約品質に 恒常的な不満が出たら → Tavily(②) に news-search-provider Parameter Store で切替（無停止）。
  • 月配信頻度が 1日1回 → 1日3回以上に増えたら → 無料枠超過のため Brave Pro（$5/月 で 20,000q）または Tavily 有料プランに移行。
