# 計画書 → タスク分解 → 実行フロー

複雑な実装計画を確実に実行するための標準フロー。

---

## フロー概要

```
1. 設計書作成
       ↓
2. レビュー（3回以上推奨）
       ↓
3. タスク分解 (TaskCreate)
       ↓
4. 依存関係設定 (TaskUpdate)
       ↓
5. 進捗ファイル作成
       ↓
6. 実行
       ↓
7. 完了確認
```

---

## Phase 1: 設計書作成

### 入力
- 要件・課題の把握

### 出力
- `docs/plans/YYYY-MM-DD-NN-{topic}.md`

### アプローチの選択

| 状況 | 推奨アプローチ |
|------|---------------|
| 要件が明確 | 直接設計書を書く |
| 要件が曖昧・大規模 | **インタビュー方式**: Claudeに質問させて要件を掘り下げ→SPEC.mdに整理→新セッションで実装 |
| 1文で差分を説明できる | 計画をスキップして直接実装 |

**インタビュー方式の使い方:**
Claudeに「この機能について詳しくインタビューして。技術実装、UX、エッジケース、トレードオフについて質問して。終わったら仕様書にまとめて」と依頼する。仕様書完成後は `/clear` して新セッションで実装に入る（クリーンなコンテキストで集中できる）。

### チェックリスト
- [ ] 目的（WHY）が明確
- [ ] 技術選定理由が明確
- [ ] コード例は「参考レベル」と明記
- [ ] 環境変数・依存関係を記載
- [ ] 各タスクの完了条件を記載

---

## Phase 2: レビュー

### レビュー観点（3回実施推奨）

| 回 | 観点 | チェック項目 |
|----|------|-------------|
| 1回目 | 実装可能性 | コード例の構文エラー、import不足、型不整合 |
| 2回目 | 設計完全性 | 欠落機能、エラーハンドリング、エッジケース |
| 3回目 | 運用・アーキテクチャ | 環境変数、デプロイ順序、ロールバック、監視 |

### Writer/Reviewerパターン（推奨）

設計書のレビューは**別セッション**で行うとバイアスが減る：

```
セッションA（設計者）: 設計書を作成
セッションB（レビュアー）: 「設計書 @docs/plans/xxx.md をレビューして。エッジケース、実装可能性、運用面を確認して」
セッションA: レビュー結果を反映
```

同じセッションで書いた直後にレビューすると、自分の設計に引きずられやすい。

### 発見事項の対応

| 種別 | 対応 |
|------|------|
| 設計書で修正すべき | その場で修正 |
| 将来対応（P2/P3） | `docs/issues/` にIssue作成 |

---

## Phase 3: タスク分解

### 原則

1. **1タスク = 1つの成果物**（ファイル or 機能単位）
2. **完了条件を明示**（チェックリスト形式）
3. **検証コマンドを記載**
4. **依存関係を明示**

### TaskCreate テンプレート

```
subject: Phase X.Y: {タスク名}
description: |
  **ファイル:**
  - 作成: `path/to/new.py`
  - 修正: `path/to/existing.py`
  - テスト: `tests/path/to/test.py`

  **完了条件:**
  - [ ] 条件1
  - [ ] 条件2
  - [ ] テスト実行

  **依存:** Task X.Y, X.Z
```

### 粒度ガイドライン

| 粒度 | 推奨 | 例 |
|------|------|-----|
| 1ファイル作成 | ○ | handler.py作成 |
| 2-3ファイル修正 | ○ | 関連ファイル3つを修正 |
| 7ファイル以上 | △ | 実行時に分割を検討 |

---

## Phase 4: 進捗ファイル作成

### 配置場所

```
.claude/execution-progress/YYYY-MM-DD-NN-{topic}.md
```

### 必須セクション

1. **実行状態サマリー**: Phase別の完了率
2. **各タスク詳細**: 状態・ファイル・依存・完了条件
3. **推奨実行順序**: 依存関係を考慮した順序図
4. **発見された課題**: 実行中に発見した問題
5. **更新履歴**: 変更ログ

---

## Phase 5: 実行

### 実行中のルール

1. **TaskUpdate で status を更新**
   - 開始時: `in_progress`
   - 完了時: `completed`

2. **1タスク1コミット**
   - 大きいタスクはサブコミットに分割

3. **進捗ファイルを同時更新**
   - チェックリストにチェック
   - 完了日時を記録

4. **課題発見時は即記録**
   - `docs/issues/` にIssue作成

---

## Phase 6: 完了確認

### 各タスク完了時

- テスト実行結果を確認
- lint/type check 通過を確認
- 進捗ファイルを更新

### 全Phase完了時

1. 進捗ファイルの全タスクが完了
2. 全テスト通過
3. INTEGRITY_CHECK.md 更新
4. PR作成

---

## チェックリスト（実行漏れ防止）

### タスク分解時
- [ ] 全タスクをTaskCreateで作成
- [ ] 依存関係をTaskUpdateで設定
- [ ] 進捗ファイルを作成

### 実行時
- [ ] 開始前に TaskList で確認
- [ ] blockedBy が空のタスクから開始
- [ ] 完了ごとに進捗ファイル更新
- [ ] 課題発見時は即Issue作成

### 完了時
- [ ] 全テスト通過
- [ ] 進捗ファイルの全チェックボックスON
- [ ] INTEGRITY_CHECK.md 更新
- [ ] PR作成
