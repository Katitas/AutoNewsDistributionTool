# セッション管理・コンテキスト運用

Claude Codeのコンテキストウィンドウは最も重要なリソース。
効率的に管理することで、長時間のセッションでも品質を維持できる。

---

## コンテキスト管理の原則

1. **1セッション = 1テーマ** — タスクが切り替わったら `/clear` でリセット
2. **調査はサブエージェントに委任** — メインコンテキストを調査結果で汚さない
3. **巨大な出力を避ける** — ログ全文よりエラー箇所の抜粋を指示

---

## セッション操作コマンド

| コマンド | 用途 | 使いどころ |
|---------|------|-----------|
| `/clear` | コンテキスト全消去 | タスク切り替え時 |
| `/compact` | 要約して圧縮 | 長時間セッションで応答が遅くなったら |

---

## サブエージェント活用パターン

大量のファイル読み込みや探索が必要な場合、サブエージェントに委任する：

| 場面 | やり方 |
|------|--------|
| コードベース調査 | 「〜を調べて」→ Agentツールで探索 |
| 複数ファイルの一括変更 | 独立した変更をサブエージェントに分散 |
| テスト実行 + 結果分析 | サブエージェントでテスト→結果サマリーのみ受け取る |

**メリット**: メインのコンテキストに探索過程が残らず、結果だけを使える

## よくある失敗パターンと対策

### 1. キッチンシンクセッション

**症状**: 1セッションで「設計して」「実装して」「テストして」「デプロイして」を全部やる
**問題**: コンテキストが枯渇し、後半で初期の設計判断を忘れる
**対策**: フェーズごとにセッションを分ける。判断結果はファイル（plans/, INTEGRITY_CHECK.md）に書き出す

### 2. 過剰修正

**症状**: 1つの修正が別の問題を生み、連鎖的に修正範囲が広がる
**問題**: 本来の目的を見失い、関係ない箇所まで変更する
**対策**: 最小差分で修正 → テスト → 次の修正。1ループ=1変更を徹底

### 3. CLAUDE.md肥大化

**症状**: 便利だからとCLAUDE.mdにルールを追加し続ける
**問題**: 毎回全文読み込まれるため、コンテキストを圧迫する
**対策**: 100行以内を維持。詳細は `.claude/rules/` に分離（`doc-governance.md`参照）

### 4. 信頼→検証ギャップ

**症状**: 「修正しました」と報告するが、実際にテストを実行していない
**問題**: 潜在バグが残り、後続タスクで発覚して手戻りが大きくなる
**対策**: 「動くはず」ではなく「動いた証拠」を残す（`execution-checklist.md`参照）

### 5. 無限探索

**症状**: 完璧な情報を得ようとファイルを読み続け、実装に着手しない
**問題**: コンテキストが探索結果で埋まり、実装時に必要な余裕がなくなる
**対策**: 探索は3回まで。3回で見つからなければ仮説を立てて進む

## セッション設計ガイドライン

### 短いセッション（推奨）

```
1セッション = 1機能 or 1バグ修正
- 計画確認 → 実装 → テスト → コミット
- セッション終了前: 「品質チェックして」で変更の影響を確認
- 所要: 15-30分
```

### 長いセッション（注意が必要）

```
1セッション = 複数機能を順次実装
- /compact を適宜実行
- 判断結果はファイルに書き出してコンテキスト外に退避
- 方針変更があればCLAUDE.mdやplans/に記録
- 各機能の区切りで「品質チェックして」を実行し、乖離の蓄積を防ぐ
```
