# ドキュメント運用ガイド

## ドキュメント構造マップ

```
docs/
├── design/           # 詳細設計書（HOW）— 正式な設計ドキュメント
├── guides/           # 手順書（HOW）
├── plans/            # 作業中の実装計画
├── issues/           # 課題・改善点の記録
│   ├── README.md     # 全Issue一覧
│   ├── critical/     # P0: 緊急（本番障害）
│   ├── high/         # P1: 高（機能影響）
│   ├── medium/       # P2: 中（改善推奨）
│   └── low/          # P3: 低（いつかやる）
└── INTEGRITY_CHECK.md # 設計書と実装の整合性チェック
```

---

## 新規ドキュメントの配置ルール

### 判断フロー

1. **実装の根拠となる仕様か？** → `design/` 配下の適切なセクション
2. **手順・やり方の説明か？** → `guides/`
3. **作業中の計画・設計か？** → `plans/`（実装完了後 `design/` へ昇格）
4. **課題・改善点か？** → `issues/`

### 命名規則

| ディレクトリ | 命名パターン | 例 |
|-------------|-------------|-----|
| design/ | `NN_{トピック}.md` | `01_認証設計.md` |
| plans/ | `YYYY-MM-DD-NN-{topic}.md` | `2026-01-28-01-feature-design.md` |
| guides/ | `{topic}.md` | `local-dev.md` |

---

## plans/ → design/ 昇格ルール

以下の条件を満たしたら `design/` へ移動：

1. 実装が完了している
2. レビューを経ている
3. 今後の実装で参照される

昇格時の作業：
- ファイル名を `design/` の命名規則に変更
- 適切なセクションディレクトリに配置
- plans/ から元ファイルを削除

## ドキュメント階層

```
要件定義（WHY/WHAT） → 詳細設計（HOW） → 実装コード
```

| 階層 | 内容 | 更新タイミング |
|------|------|---------------|
| 要件/仕様 | ビジネス要件・ユースケース | 要件変更時 |
| design/ | API仕様・データスキーマ・処理フロー | 実装変更時 |
| 実装コード | ソースコード | 随時 |

## CLAUDE.md 肥大化防止

### 本体に書いてよいもの（100行以内維持）

| 含めるべき | 理由 |
|-----------|------|
| Claudeがコードを読んでも推測できないBashコマンド | テストランナーやビルド手順は暗黙知 |
| デフォルトと異なるコードスタイルルール | 標準規約と違う部分のみ |
| テスト指示と推奨テストランナー | 検証方法はコードから分からない |
| リポジトリのエチケット（ブランチ命名、PR規約） | Git運用はコード外の規約 |
| プロジェクト固有のアーキテクチャ決定 | 「なぜこうなっているか」の説明 |
| 一般的な落とし穴・明白でない動作 | ハマりやすいポイント |

### 本体に書いてはいけないもの

| 除外すべき | 理由 |
|-----------|------|
| Claudeがコードを読めばわかるもの | 冗長。コードが正 |
| 標準的な言語規約 | Claudeは既に知っている |
| 詳細なAPIドキュメント | ドキュメントへのリンクで十分 |
| 頻繁に変わる情報 | すぐ古くなり誤誘導する |
| 長い説明やチュートリアル | `guides/` or `.claude/rules/` へ |
| ファイルごとのコードベース解説 | `project-structure.md` へ |
| 「きれいなコードを書く」等の自明な指示 | 効果なし |

### 判断基準

各行について「**これを削除するとClaudeが間違いを犯すか？**」と問う。答えがNoなら削除する。
肥大化したCLAUDE.mdは、重要なルールがノイズに埋もれてClaudeが指示を無視する原因になる。

### 新規ルール追加時のフロー

1. **毎回適用 + 違反が致命的** → CLAUDE.md本体に追記
2. **毎回適用 + ガイドライン的** → `.claude/rules/` に新規ファイル作成
3. **特定タスクで必要** → `docs/guides/` に追加
4. **ドメイン知識・再利用ワークフロー** → `.claude/skills/` にスキル作成

## ソースコードと設計書の紐づけ

### 紐づけの原則

コードのすべての機能領域は、対応する設計書を持つべきである。
この対応関係は2箇所で管理する：

1. **`project-structure.md`** の「ソースコードと設計書の対応表」— どのコードがどの設計書に対応するか
2. **`INTEGRITY_CHECK.md`** — 各設計書と実装の整合ステータス

### 新規機能追加時のフロー

```
1. docs/plans/ に計画書を作成
2. 実装
3. 実装完了後、plans/ → design/ に昇格
4. project-structure.md の対応表に行を追加
5. INTEGRITY_CHECK.md に行を追加（ステータス: ✅）
```

### 設計書のセクション設計

設計書は**ソースコードのディレクトリ構造と対応**させることを推奨する：

```
ソースコード              設計書
src/handlers/auth/   ←→  docs/design/01_認証設計.md
src/handlers/users/  ←→  docs/design/02_ユーザー管理.md
src/db/              ←→  docs/design/10_データ層.md
infra/               ←→  docs/design/11_インフラ設計.md
```

これにより「このコードの仕様はどこを見ればよいか」が即座に分かる。

## ドキュメント整合性維持

### 実装時の更新ルール

| 実装内容 | ドキュメント対応 |
|----------|-----------------|
| 設計書にない変更を実装 | 該当ドキュメントに追記・修正 |
| 仕様変更 | 設計書に変更履歴を追記 |
| 新規機能追加 | 対応する設計書セクションに追記 |
| 暫定対応 | `issues/` にIssue記録 |
| 新規コード領域追加 | `project-structure.md` の対応表に追記 |

### 整合性チェックリスト

`INTEGRITY_CHECK.md` で設計書と実装の整合性を管理する。
変更後はステータスを更新すること。

### 乖離防止の鉄則

1. **ドキュメント更新は実装と同時** — 「あとで」は禁止
2. **対応表を常に最新に** — `project-structure.md` と `INTEGRITY_CHECK.md` の行は一致させる
3. **乖離を見つけたら即対応** — 放置は技術的負債の始まり

## Issue管理

### 記録タイミング

| 発見状況 | アクション |
|----------|-----------|
| 実装中に課題発見 | 即座にIssue MDを作成 |
| 暫定対応で進めた | 本対応をIssueとして記録 |
| 将来的な改善点 | `low/` に記録 |

### Issue配置

重要度フォルダ → カテゴリフォルダ の階層構造。

```
issues/
├── high/
│   ├── api/
│   │   └── 001-issue-title.md
│   ├── infra/
│   └── docs/
...
```

### Issue MDの記載内容

Claudeがファイルを読んで以下を即座に理解できる詳細さで記載：

- **どこで**: リポジトリ、ファイルパス、行番号
- **何が**: エラーメッセージ、現象
- **なぜ**: 原因分析
- **どう対応**: 対応案と推奨

### Issue対応SLA

| 重要度 | 基準 | 対応期限 |
|--------|------|----------|
| P0 (critical) | 本番障害・セキュリティ問題 | **即時対応** |
| P1 (high) | 機能に影響がある不具合 | **1週間以内** |
| P2 (medium) | 改善推奨・技術的負債 | **次スプリント** |
| P3 (low) | いつかやりたい改善 | **バックログ** |
